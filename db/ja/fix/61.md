HTTP リクエストの密輸の脆弱性を軽減するには、次の対策を実装することを検討してください。

1. **HTTP/2 をエンドツーエンドで採用する:**
    - 可能な限り、HTTP/2 のエンドツーエンドの利用を選択し、HTTP ダウングレードを無効にします。 HTTP/2 は、リクエストの長さを決定するための堅牢なメカニズムを採用しており、リクエストの密輸に対する固有の保護を提供します。
2. **書き換えられたリクエストを検証する:**
    - HTTP のダウングレードが避けられないシナリオでは、書き換えられたリクエストが HTTP/1.1 仕様に照らして注意深く検証されます。ヘッダーに改行、ヘッダー名にコロン、リクエスト メソッドにスペースが含まれるリクエストを拒否するなど、厳密な検証基準を適用します。
3. **あいまいなリクエストを正規化する:**
    - フロントエンド サーバーが曖昧なリクエストを正規化し、バックエンド サーバーが曖昧なままのリクエストを拒否し、その後 TCP 接続を閉じる戦略を実装します。これにより、2 つのサーバー間のリクエスト境界を一貫して理解できるようになります。
4. **リクエストボディの説明:**
    - この仮定は CL.0 とクライアント側の非同期の両方の脆弱性の根底にあるため、リクエストに本文が含まれないという仮定は避けてください。基本的な予防策として、潜在的なリクエスト本文の存在に注意してください。
5. **サーバー例外で接続を破棄:**
    - リクエストの処理中にサーバーレベルの例外がトリガーされた場合、デフォルトでは接続が破棄されます。このアプローチにより、例外が発生したときに発生する可能性のある脆弱性が悪用されるリスクが最小限に抑えられます。
6. **フォワード プロキシのアップストリーム HTTP/2 を有効にする:**
    - トラフィックがフォワード プロキシ経由でルーティングされる場合は、可能な限りアップストリーム HTTP/2 が有効になっていることを確認してください。これは、リクエスト処理の一貫性を維持し、リクエストの密輸の脆弱性の可能性を減らすのに役立ちます。
